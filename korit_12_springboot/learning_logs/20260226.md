# SpringBoot 3- Tier Architecture
- SpringBoot 프로젝트상에서 애플리케이션의 유지보수성과
확장성을 높이기위해서 역할을 분리하는 표준적인 설계방식
방식으로 각 계층은 고유한 책임을가진다. 이 계층사이를 이동할때 
DTO를 통해 이동

- 만약 Entity class User 에 id/ username/ password/ name/ address 라고하는 필드들이 있는데 . postman을 통해 

{"username":"user", "password":"password"}가 전송됐다고했을때
이는 Controller를 통해들어오게되고
Repository에서 확인하게된다. 
- postman에서 데이터를 입력받으면 controller에서 등록,조회하고
UserReposiotry의 findByUsername을 호출해서 특정 User객체가 return된다. 그 후에 service계층에서 리턴된 User객체가 isComplete 가 적용된다.
그리고 각 로직에맞게(post,get)수행되고 repository에 데이터를담아 
Entity객체를 DB테이블과 매핑하고 실제 SQL을 실행하여 데이터를 처리한다. 그리고 다시 service로와서 get요청을받은것을 controller에서 응답받는다 그걸최종적으로 postman에서출력

# 종류

1. Controller (Presentation Layer): 클라이언트의 요청을 가장 먼저받는 관문역할, 요청데이터를 검증하고 , 비즈니스 로직을 수행하기 위해 서비스 계층을 호출하며 최종결과를 응답으로 반환한다
그래서 처음에 요청을받는역할할때 한번 마지막에 응답을 반환할때한번 2번 실행된다.

2. service(Business Logic Layer) : 애플리케이션의 핵심 비즈니스 로직이 수행되는 곳이다
데이터의 가공, 트랜잭션관리, 앤티티와 DTO간의 변환등을 담당한다.

3. Repository (*DB에서의 각 테이블 ROW
 Data Access Layer): 데이터베이스에 직접 접근하는 계층이다. JPA를 사용했으니 Entity객체를 DB테이블과 매핑하고 실제 SQL을 실행하여 데이터를 처리한다
- POSTMAN에서 요청이 들어오면 Controller에서 요청을 검증하고 ->Service 계층으로 넘겨서 특정정보가있는지 체크하고-> Repository
에서 가져온다음 -> Service계층에서 혹시 null값이 있는지 아니먄 정보조합이되는지 여부를 따지고  ->Controller로 보내서
올바르다면 postman에제대로띄울것 아니면 에러

- postman-> controller->service->repository->db(여기까지요청)->
service->controller(여기가응답객체리턴)->postman
- 요청: Postman ➔ Controller ➔ Service ➔ Repository ➔ DB

처리: DB에서 꺼낸 데이터가 Entity에 담겨 Repository ➔ Service로 복귀

가공: Service에서 isComplete 변경 등 로직 수행

응답: Service ➔ Controller ➔ Postman (최종 출력)
4. DTO (Data Transfer Object)
- 계층 간 데이터 이동시에 데이터베이스 앤티티를 직접노출하지않고 DTO 를 사용한다.
- 보안 :DB 구조가 변경되어 엔티티가 수정되더라도 API 스펙(DTO)은 유지할 수 있습니다.
- 결합도 ⬇️ :DB구조가 변경되어 앤티티가 수정되더라도 API스팩은유지
- 유효성검사: 클라이언트가 보내는 데이터에 대해 특정계층에서만 필요한 검증로직을 수행하기에 유용

  - DTO의 경우 request 버전과 responce버전으로 나뉘는 경우가있다
  보낼떄는 username/password였지만 돌아올떈 username/ password/todos혹은 cars등이있다.

5. service내에 todoservice 




```java
package com.todo.todolist.service;


import com.todo.todolist.dto.TodoRequest;
import com.todo.todolist.dto.TodoResponse;
import com.todo.todolist.entity.Todo;
import com.todo.todolist.entity.User;
import com.todo.todolist.repository.TodoRepository;
import com.todo.todolist.repository.UserRepository;
import lombok.RequiredArgsConstructor;
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

import java.util.List;
import java.util.stream.Collectors;

@Service
@RequiredArgsConstructor
public class TodoService {
    private final TodoRepository todoRepository;
    private final UserRepository userRepository;
    
    //post요청을 하는 비즈니스 로직작성
    @Transactional
    public TodoResponse createTodo(Long userId, TodoRequest request){
        User user = userRepository.findById(userId).orElseThrow(()->new RuntimeException("사용자찾을수없음"));

        Todo todo = new Todo(request.content(), user); // Todo객체 만들어지면isComplete 가 적용된다.
        Todo saveTodo = todoRepository.save(todo); 
        
        
        // Entity를 dto로 변환해서 Controller 로 넘겨주기
        return new TodoResponse(saveTodo.getId(), saveTodo.getContent(),saveTodo.isCompleted());
    }
    
    
    //get 요청하는로직
    @Transactional(readOnly = true)
    public List<TodoResponse> getTodoList(){
        // list에 하나하나 새로운함수를 적용 map()
        return  todoRepository.findAll().stream()
                .map(todo -> new TodoResponse(todo.getId(), todo.getContent(),todo.isCompleted()))
                .collect(Collectors.toList());        
        
    }
}


```

다양한 repository를 field로 삼아서 메서드들을 호출한다음, 필요한
정보들을 가공해서 TodoResponse 객체로 return했다는점을 확인할수있다.
내부로직 중에 제어문의 다수가 쓰였다는것 
lamda expression 이 쓰이고있다는 점에서 이전 java에서 배경지식이요구된다. 그리고  .map 복습이요구


6. Controller 패키지에 todoController생성

```java
package com.todo.todolist.controller;

import com.todo.todolist.dto.TodoRequest;
import com.todo.todolist.dto.TodoResponse;
import com.todo.todolist.service.TodoService;
import lombok.RequiredArgsConstructor;
import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.*;

import java.util.List;

@RestController
@RequestMapping("/api/todos")
@RequiredArgsConstructor

public class todoController {
    private final TodoService todoService;

    // todo 등록 api
    @PostMapping("/{userId}")
    public ResponseEntity<TodoResponse> addTodo(@PathVariable Long userId, @RequestBody TodoRequest request) {
        // 결과를response 객체명에저장햇다
        TodoResponse response = todoService.createTodo(userId, request);
        return new ResponseEntity<>(response, HttpStatus.CREATED);

    }
    // todo 조회 api
    @GetMapping
    public ResponseEntity<List<TodoResponse>> getALLTodos(){
        return ResponseEntity.ok(todoService.getTodoList());
    }
}

```
- basePath = api로 해놓은 부분을 Controller 클래스에서 세팅해놓은것을 본다.
그래서 @RestMapping 뿐 아니라 @RequestMapping("/api/todos")가 따로 존재한다.

-postmapping {userId}부분까지 합쳤을떄 걀과적으로 http://localhost:8080/api/todos/{userId}라고할수있다.

 - {userId} 의 {}는 endpoint상에서 달라질수있는 영역을 동적으로 받아와서 addTodo()메서드의 매개변수로 쓰겠다는것을의미
 1. '@RequestBody'
 - HTTP요청 메서드(POST, PUT, PATCH)의 요청 본문에 담긴 데이터를 읽어와 SpringController 메서드의 메게변수로 연결할때 사용한다.
 클라이언트의 서버로 데이터를 전송할때 주로사용한다. 
 즉 postman상에서 body란에 작성한 내용이 addTodo() 매개변수로 사용되었음, 그리고 그 형태가 TodoRequest자료형이었다.
  2. 특징 및 작동원리
 - 자동 역직렬화 : 
 (제이슨에서 자바객체로 바꾸는것)
 XML/JSON 등의 데이터를 메서드 매개변수로 지정된 JAVA객체로 자동반환해준다. 
 즉 @RequestBody에너테이션으로인해 TodoRequest로 변환되었다.

- 메시지 컨버터: Spring 의 'HttpMessageConverter'가 담당하는 부분으로 json데이터가 들어오면 jackson라이브러리를 사용하는 컨버터가
이를 java 객체로 변환

- 데이터 바인딩: 


- 참고: 요청 본문이 없는 get요청에는 사용할수없고 보통은 'post','put'과 같이 데이터전송이 필요한 요청에 사용된다.

3. @PathVariable
- 해당 에너테이션은 Spring Web 에서 URL 경로 변수에 있는 값을 추출하여 Controller 의 메서드의 매개변수로 바인딩하는데 사용된다. 
이는 특정 리소스를 식별하거나 
**조회**할때 RESTfUL API 디자인을 구현하는데있어 핵심.
- 특징 및 작동원리
* URL 경로 매핑: URL 경로의 특정 부분을 변수처럼`{id}`와 같이표시하고 , 그 변수에 할당된 실제 값을 메서드의 매개변수로 사용
*  RESTfUL식별자 :주로 개별 리소스(특정 사용자 ID/ 특정 상품번호)
를 식별하기 위해 사용되며 URL 의 의미를 명확히 만들어줌

* 자동타입반환 : 경로에서 추출된 문자열은 메서드의 매개변수타입에맞게(Long, Integer, String) 자동반환
